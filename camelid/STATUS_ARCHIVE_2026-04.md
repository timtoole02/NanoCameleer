This file is Camelid's evidence ledger for current support claims. `README.md` frames the product, `COMPATIBILITY.md` defines the release contract, `ROADMAP.md` carries sequencing, and this file records the artifacts and validation behind those claims.

## Executive snapshot

Camelid is in controlled expansion mode. Public support remains intentionally narrow: one supported generation gate, one narrow evidence row, one exact acceptance target, and one larger-model groundwork lane. The supported gate is **TinyLlama 1.1B Chat Q8_0**: the five-prompt, 50-token TinyLlama parity audit matches known-good llama-server prompt tokens, generated token arrays, and generated text. The token-major `output.weight` layout fix remains the core correctness boundary and must stay protected across ports, optimizations, and packaging work.

Current adjacent-lane position:

- **Llama 3.2 1B Instruct Q8_0:** narrow small-model evidence exists for one compact-header `hello` prompt through five deterministic tokens (`[9906,0,2650,649,358]` / `"Hello! How can I"`). Useful evidence; not broad Llama 3 support.
- **Llama 3.2 3B Instruct Q8_0:** the exact WebUI acceptance GGUF is now present in the model directory and metadata/API model load works. Recent file-backed lazy-Q8 recovery work removed most of the old eager dense-load spike, but the guarded first-chat retry still stopped before any token; no 3B generated-token, parity, or WebUI evidence exists yet.
- **Llama 3 8B Instruct Q8_0:** metadata/config/tokenizer/template fixtures, independent tokenizer references, Q8_0 block-only loading, and serial row/all-row dot groundwork exist. Generation remains unsupported until lazy/mmap/on-demand Q8_0 linear execution is wired through attention, FFN, and output projection and QA records bounded first-token memory/parity evidence.
- **Frontend/API support contract:** user-visible readiness requires both runtime readiness (`loaded_now=true`, `generation_ready=true`) and an exact supported compatibility row. Same-family names or paths must not inherit support across quantization or model-size boundaries.

Next support-changing milestone: land lazy/mmap/on-demand Q8_0 linear execution through token embedding/output projection, attention, and FFN, then rerun bounded first-token memory/parity evidence for the blocked Llama 3 generation lanes.

Operating rule: keep support claims exact-row, evidence-backed, aligned across `README.md`, `COMPATIBILITY.md`, `ROADMAP.md`, `/api/capabilities`, and frontend copy, and preserve visible llama.cpp / ggml MIT credit wherever reference tooling or parity evidence depends on it.

## Latest backend Llama 3.2 3B lazy-Q8 recovery follow-up

Backend kept the lane serialized, found no running llama-server/backendinference serve process, and re-read `STATUS.md` / `ROADMAP.md` before changing runtime code. The exact 3B RAM gap is now pinned: llama-server mmap/streams Q8 weights, while Camelid's old first-generation path entered `LlamaLoadedWeights::load` and eagerly decoded Q8_0 tensors into dense f32; earlier local fixes removed the tied-output clone and default retained-Q8 duplication, but still left ~12 GiB of dense materialization. This follow-up adds an opt-in `BACKENDINFERENCE_LAZY_Q8_0_LINEAR=1` path that leaves Q8_0 linear tensors file-backed instead of f32-materialized, routes attention/FFN/output projection through existing Q8 block readers, and adds file-backed Q8 embedding lookup so the tied `token_embd.weight` no longer has to become a 1.468 GiB f32 tensor just to embed prompt tokens or serve as the output projection.

Validation passed `cargo test q8_0_file_backed -- --nocapture`, `cargo test q8_0_block_reader_linear_matches_existing_q8_path -- --nocapture`, `cargo test --test inference_session -- --nocapture`, `cargo clippy --all-targets --all-features -- -D warnings`, `cargo test --all-targets --all-features`, `cargo build --release --bin backendinference`, `cargo fmt --all -- --check`, and `git diff --check`. One bounded 3B smoke was attempted with no llama-server and `BACKENDINFERENCE_LAZY_Q8_0_LINEAR=1` under `target/llama32-3b-lazy-q8-smoke-20260430T213910Z/`: `/api/models/load` succeeded and health reported `generation_ready=true`, but the conservative safety guard killed the server before any token when free+speculative+purgeable memory reached `65,435` 16 KiB pages (~1.0 GiB) and RSS was ~2.5 GiB. That is not a successful 3B token artifact and should not promote support; it is evidence the eager dense-load spike is mostly removed, with the next retry needing a quiet-memory window and the same single-run guard rather than parallel parity loops.
